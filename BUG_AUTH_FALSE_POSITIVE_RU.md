# Баг: ложноположительная проверка авторизации Claude CLI

## Проблема
Приложение сообщает, что AI-авторизация подтверждена, хотя Claude CLI внутри контейнера фактически не залогинен.

## Симптомы
После запуска сервиса в логах появляется сообщение:

```text
[startup] Auth confirmed. Mounting webhook handler.
```

Но реальная проверка внутри контейнера показывает отсутствие авторизации:

```bash
docker compose exec -T review-bot claude auth status
```

Результат:

```json
{
  "loggedIn": false,
  "authMethod": "none",
  "apiProvider": "firstParty"
}
```

## Причина
В `src/ai/claudeProvider.ts` метод `checkAuth()` использовал несовместимый вызов:

```ts
spawn('claude', ['auth', 'status', '--output-format', 'json'], { stdio: 'pipe' })
```

В текущей версии Claude CLI опция `--output-format` для `claude auth status` не поддерживается, команда завершается с ошибкой, и приложение ошибочно считает, что авторизации нет.

## Почему это баг
Из-за этого система:

1. Пишет в логах неверную информацию.
2. Раньше времени монтирует webhook handler.
3. Создаёт ложное ощущение готовности сервиса.
4. Может привести к сбоям уже во время реального review.

## Ожидаемое поведение
Проверка авторизации должна подтверждаться только если Claude CLI действительно залогинен.

## Предлагаемое исправление
Заменить проверку `claude --version` на реальную проверку статуса авторизации, например:

```bash
claude auth status
```

И считать авторизацию подтверждённой только если:

1. команда завершилась с кодом `0`
2. в ответе явно указано `loggedIn: true`

## Затронутые файлы
- `src/ai/claudeProvider.ts`
- `src/ai/authChecker.ts`
- `src/index.ts`

## Приоритет
Высокий, потому что баг вводит в заблуждение при прод-запуске и диагностике.
